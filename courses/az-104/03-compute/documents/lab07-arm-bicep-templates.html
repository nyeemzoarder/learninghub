Lab 07 – ARM Templates & Bicep

⏱️ Total Estimated Time: 55 minutes

Real-Life Scenario

Company: TechCorp Inc

TechCorp is deploying applications to Azure using Infrastructure as Code (IaC). Manual deployments are error-prone and inconsistent. Your infrastructure team needs to automate infrastructure deployment using Bicep templates, replicate deployments consistently across dev/test/production, preview changes before applying them, and track deployment history for auditing.

The Challenge: Can you create a reusable Bicep template that deploys a consistent infrastructure stack across all environments?

❗

Core Learning Objective

Infrastructure as Code (IaC) transforms infrastructure into version-controlled, reproducible code. Bicep is Azure's modern templating language—simpler and more readable than ARM JSON but equally powerful.

Objectives

  • Author Bicep templates with parameters and variables
  • Use what-if deployments to preview changes safely
  • Deploy resources at resource group scope
  • Understand Incremental vs Complete deployment modes
  • View and troubleshoot deployment history

Prerequisites

  • Azure CLI with Bicep installed: az bicep install
  • Signed in to Azure: az login

Part 2 – Author a Bicep template

⏱️ 15 minutes
ℹ️

Bicep Basics

Bicep is a domain-specific language (DSL) that compiles to ARM JSON. It offers cleaner syntax with parameters (inputs), variables, resources, and outputs.

✓

Step 1: Create main.bicep File

  1. Create a new directory for this lab (e.g., C:\Lab07) — avoid spaces in the path
  2. In that directory, create a file named main.bicep with the following content:
@description('Name prefix for resources')
param namePrefix string = 'az104lab07'

@allowed(['Standard_LRS', 'Standard_GRS'])
param storageSku string = 'Standard_LRS'

var storageAccountName = toLower('${namePrefix}${uniqueString(resourceGroup().id)}')

resource storageAccount 'Microsoft.Storage/storageAccounts@2023-01-01' = {
  name: storageAccountName
  location: resourceGroup().location
  sku: {
    name: storageSku
  }
  kind: 'StorageV2'
  properties: {
    minimumTlsVersion: 'TLS1_2'
    allowBlobPublicAccess: false
  }
}

resource vnet 'Microsoft.Network/virtualNetworks@2023-09-01' = {
  name: '${namePrefix}-vnet'
  location: resourceGroup().location
  properties: {
    addressSpace: {
      addressPrefixes: ['10.20.0.0/16']
    }
    subnets: [
      {
        name: 'default'
        properties: {
          addressPrefix: '10.20.0.0/24'
        }
      }
    ]
  }
}

output storageAccountName string = storageAccount.name
output vnetId string = vnet.id
  1. Save the file in your lab directory
ℹ️

Bicep Structure Explained

param: Input parameters (like function arguments). var: Variables computed from parameters. resource: Azure resources to deploy. output: Values returned after deployment.

✓

Step 2: Verify Bicep Syntax

  1. In your terminal, navigate to the lab directory where you saved main.bicep
  2. Run the build command to check for syntax errors:
az bicep build --file main.bicep
  1. If successful, a main.json (ARM JSON) file will be created automatically
  2. If there are errors, fix them in the .bicep file and try again

Part 2 Validation

After Part 2

  • You authored a Bicep template with parameters, variables, and resources
  • You understand the structure of Bicep files (params, vars, resources, outputs)
  • You can compile Bicep to ARM JSON
  • You know how uniqueString() generates deterministic names

Part 3 – Preview with what-if

⏱️ 8 minutes
❗

What-if Deployments

Always run what-if before deploying to production. It shows exactly what changes will occur (Create, Modify, Delete, NoChange) without modifying any resources.

✓

Step 1: Run What-if Deployment Preview

  1. In your terminal, from the lab directory, run the what-if command:
az deployment group what-if \
  --resource-group rg-az104-lab07 \
  --template-file main.bicep
  1. Review the output carefully. You should see:
  • Create: Storage account (with the generated unique name)
  • Create: Virtual Network az104lab07-vnet
  1. Verify that the predicted resources match your template intentions before proceeding

Part 3 Validation

After Part 3

  • You ran a what-if deployment to preview changes
  • You reviewed predicted Create operations before deploying
  • You understand what-if is a safe way to validate templates

Part 4 – Deploy

⏱️ 10 minutes
ℹ️

Deployment Command

The az deployment group create command deploys a template to a resource group. You can override parameters on the command line or use a parameters file.

✓

Step 1: Deploy the Bicep Template

  1. From the lab directory, run the deployment command:
az deployment group create \
  --resource-group rg-az104-lab07 \
  --template-file main.bicep \
  --parameters storageSku=Standard_LRS \
  --name lab07-deployment
  1. Wait for the deployment to complete (typically 1-2 minutes)
  2. Once complete, you should see deployment status as "Succeeded"
✓

Step 2: View Deployment Outputs

  1. Query the deployment outputs with this command:
az deployment group show -g rg-az104-lab07 -n lab07-deployment --query properties.outputs
  1. You should see the storageAccountName (unique name generated by uniqueString()) and vnetId (full ARM ID)

Part 4 Validation

After Part 4

  • Bicep template successfully deployed to Azure
  • Storage account and VNet created with correct properties
  • Deployment outputs retrieved showing resource names and IDs
  • You can deploy Infrastructure as Code templates programmatically

Part 5 – Equivalent ARM JSON (for recognition on the exam)

⏱️ 7 minutes
ℹ️

Bicep vs ARM JSON

Bicep is syntactic sugar that compiles to ARM JSON. The exam may show ARM JSON syntax, so understanding the equivalence is important.

✓

Step 1: View the Generated ARM JSON

  1. Open the main.json file that was generated when you ran az bicep build
  2. Note the top-level structure:
  • $schema: URL pointing to the ARM template schema
  • contentVersion: Version number (typically "1.0.0.0")
  • parameters: Input parameters (like namePrefix, storageSku)
  • variables: Computed values (like storageAccountName)
  • resources: Array of Azure resources to deploy
  • outputs: Values returned after deployment
  1. Compare the Bicep syntax with the JSON syntax side-by-side
✓

Step 2: Understand the Mapping

  1. In the JSON, find the Storage Account resource object and note:
  • type: Microsoft.Storage/storageAccounts
  • apiVersion: 2023-01-01
  • name: References the storageAccountName variable using [variables('storageAccountName')]
  • properties: Nested object with SKU, kind, and other properties
  1. Observe that Bicep's resource block compiles to this verbose JSON structure

Part 5 Validation

After Part 5

  • You understand Bicep compiles to ARM JSON
  • You recognize ARM JSON structure ($schema, contentVersion, parameters, variables, resources, outputs)
  • You can map Bicep syntax to ARM JSON equivalents
  • You're prepared to recognize ARM JSON syntax on the exam

Part 6 – Deployment modes

⏱️ 8 minutes
⚠️

Complete Mode is Dangerous

Complete mode deletes resources in the resource group that are NOT defined in your template. Use only when you fully understand the consequences. Incremental (default) is much safer.

✓

Step 1: Understand Deployment Modes

  1. Review the two deployment modes:
  • Incremental (default): Adds/updates resources defined in the template. Leaves other resources in the RG untouched.
  • Complete: Replaces the entire resource group with only what's in the template. Deletes anything NOT in the template.
  1. Our current deployment used Incremental (the default)
✓

Step 2: Redeploy with Updated Parameter

  1. Modify the storageSku parameter to GRS and redeploy with Incremental mode:
az deployment group create \
  --resource-group rg-az104-lab07 \
  --template-file main.bicep \
  --parameters storageSku=Standard_GRS \
  --mode Incremental \
  --name lab07-deployment-v2
  1. Wait for the deployment to complete
  2. The Storage account SKU will be updated to GRS while preserving all other resources
❗

Exam Tip on Complete Mode

The exam frequently tests the difference between Incremental and Complete modes. Remember: Incremental is safe (adds/updates only), Complete is destructive (deletes resources not in template). Always use Incremental unless you explicitly want to remove resources.

Part 6 Validation

After Part 6

  • You understand the difference between Incremental and Complete modes
  • You know Complete mode is dangerous and deletes resources
  • You successfully redeployed with updated parameters
  • You can explain deployment mode tradeoffs on the exam

Part 7 – Deployment history & troubleshooting

⏱️ 7 minutes
ℹ️

Audit and Debug Deployments

Azure maintains a complete audit trail of all deployments. You can view deployment history, operation details, and error messages for troubleshooting.

✓

Step 1: View Deployment History

  1. List all deployments in the resource group:
az deployment group list -g rg-az104-lab07 -o table
  1. You should see at least two deployments:
  • lab07-deployment (Incremental, Standard_LRS)
  • lab07-deployment-v2 (Incremental, Standard_GRS)
  1. Note the timestamps showing when each deployment occurred
✓

Step 2: View Deployment Operations

  1. For the most recent deployment, view all operations:
az deployment operation group list \
  --resource-group rg-az104-lab07 \
  --deployment-name lab07-deployment-v2 \
  -o table
  1. You should see operations for each resource (Storage account, VNet) with status "Succeeded"
  2. If a deployment failed, this view would show error details
✓

Step 3: Inspect a Specific Operation

  1. Get details of a specific operation (e.g., the Storage account):
az deployment operation group show \
  --resource-group rg-az104-lab07 \
  --deployment-name lab07-deployment-v2 \
  --operation-id "0" \
  --query properties
  1. Review the output to understand resource creation details

After Part 7

Exam Tips

  • Bicep vs ARM JSON: Bicep compiles to ARM JSON. Both accomplish the same deployments; Bicep is just more readable.
  • What-if: Always shows Create/Modify/Delete/NoChange operations—use it before production deployments to prevent surprises.
  • Incremental vs Complete: Incremental (default) is safe—adds/updates only. Complete is dangerous—deletes resources not in the template. Exam loves to test this distinction.
  • uniqueString(): Generates a unique, deterministic string from the resource group ID. Perfect for globally-unique names like storage accounts.
  • Parameters and Variables: Parameters are inputs (specified at deploy time), variables are computed from parameters and constants.
  • Deployment Modes: Understand when to use each mode; Complete mode is a critical exam topic.
