Hub-Spoke Topology: Enterprise Network Architecture

The scalable pattern enterprises use to centralize shared services and connect dozens of VNets
рҹҹЎ Intermediate вҸұ 20 Minutes рҹ“ҳ AZ-104 В· Module 04
Why This Matters

Imagine managing 50 separate office buildings, each with its own security guard, firewall, and reception desk. Nightmare, right? Hub-spoke is how Azure organizations scale. One central hub handles shared services (firewall, VPN, DNS) and spokes connect to it like branches to a tree. Every major enterprise uses this pattern. Understanding it unlocks most exam scenarios.

The Simple Idea

What Is Hub-Spoke?

Hub-Spoke = A network architecture where one central VNet (hub) holds shared services, and multiple VNets (spokes) peer to it вҖ” spokes don't talk directly to each other, they route through the hub.

What Goes in the Hub?

Hub VNet contains shared infrastructure:

  • VPN Gateway (on-premises connection)
  • ExpressRoute Gateway (dedicated circuit)
  • Azure Firewall (centralized inspection)
  • DNS Resolver (shared DNS)
  • Network NVA (network appliance)
Airline Network (Hub-Spoke): Old Model (Messy): NYC вҶ” LA вҶ” Chicago вҶ” Denver вҶ” Boston в””в”Җ Every city connected to every other city в””в”Җ Too many routes, hard to manage, expensive вң— Hub-Spoke Model (Clean): в”Ңв”Җв”Җв”Җ NYC (Spoke) в”Ӯ Dallas (Hub) в”Җв”Җв”Җ LA (Spoke) в”Ӯ в”ңв”Җ Chicago (Spoke) в”Ӯ в””в”Җ Boston (Spoke) All flights go through Dallas hub в”ңв”Җ Easier to manage вң“ в”ңв”Җ Centralized security/customs вң“ в”ңв”Җ Economies of scale вң“ в””в”Җ Spokes don't need direct routes вң“

Common Mistakes (What NOT to Do)

вқҢ Mistake 1: Peering All Spokes to Each Other

Wrong

Create peerings: в”ңв”Җ Hub вҶ” Prod в”ңв”Җ Hub вҶ” Dev в”ңв”Җ Prod вҶ” Dev (DIRECT PEERING!) в”ңв”Җ Prod вҶ” Test в”ңв”Җ Dev вҶ” Test в””в”Җ Too many connections to manage вң— Problems: в”ңв”Җ Lost central inspection (ProdвҶ’Dev bypasses firewall) в”ңв”Җ Hard to scale (add 4th spoke = 3 new peerings) в”ңв”Җ Security risk (no firewall filter) в””в”Җ Maintenance nightmare вң—

Fix

Only peer spokes to hub: в”ңв”Җ Hub вҶ” Prod вң“ в”ңв”Җ Hub вҶ” Dev вң“ в”ңв”Җ Hub вҶ” Test вң“ в””в”Җ Spokes don't talk directly All spoke-to-spoke traffic routes through firewall вң“

Why it fails: Defeats the purpose of hub-and-spoke architecture.

вқҢ Mistake 2: Forgetting Gateway Transit Settings

Wrong

Peering created: Hub вҶ” Prod в”ңв”Җ "Allow gateway transit": NOT enabled on hub в”ңв”Җ "Use remote gateway": Enabled on prod в”ңв”Җ Prod VM tries to reach on-premises в”ңв”Җ Traffic routes to hub, but hub can't в”Ӯ forward to VPN в”ңв”Җ Result: Connection fails вң—

Fix

Peering: Hub вҶ” Prod в”ңв”Җ Hub side: "Allow gateway transit" = YES в”ңв”Җ Prod side: "Use remote gateway" = YES в”ңв”Җ Both must be set correctly в”ңв”Җ Now prod can use hub's VPN вң“

Why it fails: Both sides of the peering need compatible settings.

вқҢ Mistake 3: Non-Transitive Routing

Wrong

Assumption: "If Hub вҶ” Prod and Hub вҶ” Dev are peered, Prod and Dev can talk" Reality: в”ңв”Җ Peering is non-transitive в”ңв”Җ A вҶ” B and B вҶ” C doesn't mean в”Ӯ A вҶ” C automatically вң— в”ңв”Җ Prod can reach Hub в”ңв”Җ Dev can reach Hub в”ңв”Җ But ProdвҶ’Dev traffic doesn't в”Ӯ automatically work в””в”Җ Need explicit UDR routing through firewall вң“

Fix

Configure UDRs in spokes: в”ңв”Җ Prod Route Table: 10.2.0.0/16 вҶ’ Firewall в”ңв”Җ Dev Route Table: 10.1.0.0/16 вҶ’ Firewall в”ңв”Җ Now traffic goes: Prod вҶ’ Firewall вҶ’ Dev в”ңв”Җ Firewall can inspect and allow/block в””в”Җ Transitive connectivity achieved вң“

Why it fails: Peering doesn't transitively connect spokes.

вқҢ Mistake 4: Overlapping Address Spaces

Wrong

Hub: 10.0.0.0/16 Prod: 10.0.0.0/16 (SAME!) Dev: 10.2.0.0/16 Try to peer: в”ңв”Җ Hub вҶ” Prod: FAILS (overlapping) вң— в”ңв”Җ Hub вҶ” Dev: OK в””в”Җ Cannot achieve full hub-spoke вң—

Fix

Plan CIDR ranges upfront: в”ңв”Җ Hub: 10.0.0.0/16 в”ңв”Җ Prod: 10.1.0.0/16 вң“ в”ңв”Җ Dev: 10.2.0.0/16 вң“ в”ңв”Җ Test: 10.3.0.0/16 вң“ в”ңв”Җ On-prem: 192.168.0.0/16 в””в”Җ All peering works вң“

Why it fails: Overlapping addresses cause routing ambiguity.
