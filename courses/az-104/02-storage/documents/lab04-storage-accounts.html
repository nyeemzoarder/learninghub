Lab 04 – Storage Accounts

⏱️ Total Estimated Time: 40 minutes

Real-Life Scenario

Company: Contoso Ltd

Contoso is a mid-sized organization that needs to consolidate file storage, archive old data, and control access to sensitive customer information. Currently, files are scattered across personal drives, USB sticks, and email attachments—creating security and compliance risks.

The Challenge:

  1. Set up a centralized Azure Storage Account with proper redundancy for disaster recovery
  2. Organize data using containers, file shares, queues, and tables
  3. Control access using access keys, SAS tokens, and Microsoft Entra ID
  4. Enforce network boundaries to prevent unauthorized access
  5. Copy and manage data using the Storage browser
❗

Core Learning Objective

Storage Accounts are the foundation of Azure data storage. You'll learn how redundancy patterns protect data, how access methods (keys vs. SAS vs. Entra) secure it, and how network rules enforce boundaries. This is critical for any Azure role.

Objectives

  • Create storage accounts with different redundancy/performance tiers
  • Configure access keys, SAS tokens, and network rules
  • Work with blob containers, queues, tables, and file shares
  • Move/copy data using the Portal's Storage browser

Prerequisites

  • A resource group (we'll create one)
  • Signed in at portal.azure.com

Part 1 – Create the resource group and storage account

⏱️ 15 minutes
ℹ️

What You're Doing

You'll create a resource group to organize resources, then set up your first storage account with LRS redundancy. This is the foundation for all blob, queue, table, and file share operations.

✓

Step 1: Create Resource Group

  1. Search for Resource groups > Create
  2. Name: rg-az104-lab04, Region: East US
  3. Review + create > Create
✓

Step 2: Create Storage Account

  1. Search for Storage accounts > Create
  2. Fill in the Basics tab with these settings:
  • Resource group: rg-az104-lab04
  • Storage account name: stoaz104lab (3–24 chars, lowercase + numbers only)
  • Region: East US
  • Performance: Standard
  • Redundancy: Locally-redundant storage (LRS)
  1. Leave Advanced defaults (StorageV2, Hot tier)
  2. Review + create > Create
❗

Important: Global Uniqueness

Storage account names must be GLOBALLY unique across all Azure customers. If your first choice is taken, add numbers until it's unique.

After Part 1

  • Resource group rg-az104-lab04 created in East US
  • Storage account created with StorageV2, LRS, Hot tier
  • You can access the storage account in the Azure Portal

Part 2 – Explore redundancy & performance options

⏱️ 10 minutes
ℹ️

Understanding Redundancy

Redundancy determines how many copies of your data exist and where. LRS is cheap (1 datacenter), while GRS protects across regions. You'll explore the options your storage account supports. Click the 🔍 magnifying glass above to zoom in on the diagram.

Redundancy Options Comparison:

Option Copies Locations Cost Best For
LRS 3 1 Datacenter $ Dev/Test
ZRS 3 3 AZs (1 region) $$ Production (regional)
GRS 6 2 Regions $$$ Disaster recovery
✓

Step 1: View Redundancy Options

  1. Open the storage account > Redundancy (under Data management)
  2. Note the available options: LRS, ZRS, GRS, GZRS, RA-GRS, RA-GZRS
  3. Observe that changing redundancy is easy, but note the cost implications
✓

Step 2: Understand Performance Tiers

  1. Go to Configuration > Access tier
  2. Review these access tiers:
  • Hot: Frequent access (cheapest storage, expensive per-operation)
  • Cool: Infrequent access (cheaper per-operation, expensive storage)
  • Archive: Long-term (cheapest storage, slow retrieval)
ℹ️

Access Tier Tip

Changing access tier between Hot/Cool is free and instant. Archive requires rehydration (24-48 hours).

After Part 2

  • You understand redundancy options (LRS, ZRS, GRS)
  • You know the cost/resilience tradeoffs
  • You can navigate to redundancy and access tier settings

Part 3 – Containers, blobs, queues, tables, file shares

⏱️ 8 minutes
ℹ️

Storage Account Data Types

Storage accounts can hold different data types: blobs (files), queues (messages), tables (NoSQL), and file shares (SMB). You'll create one of each to understand the options.

✓

Step 1: Create and Upload to Blob Container

  1. Open the storage account > Containers (under Data storage)
  2. Select + Container, name: demo-container, Public access level: Private
  3. Open demo-container > Upload
  4. Create a small text file locally (e.g., sample.txt with text "hello az104")
  5. Upload it to the container
⚠️

Warning: Public Access

Setting Public access to "Blob" or "Container" allows anonymous download. Only do this for non-sensitive data.

✓

Step 2: Create Queue and Table

  1. Queues (under Data storage) > + Queue, name: demo-queue
  2. Tables (under Data storage) > + Table, name: demotable
ℹ️

Data Types Explained

Queues = async messaging. Tables = NoSQL key-value store. File Shares = SMB protocol (like network drives).

✓

Step 3: Create File Share

  1. File shares (under Data storage) > + File share
  2. Configure with these settings:
  • Name: demo-share
  • Quota: 5 GiB
  • Tier: Transaction optimized
  1. Click Create

After Part 3

  • Container demo-container created with sample blob
  • Queue demo-queue created
  • Table demotable created
  • File share demo-share created with 5 GiB quota

Part 4 – Access keys vs SAS vs Microsoft Entra authorization

⏱️ 7 minutes
❗

Three Ways to Access Storage

Access Keys: Full admin privileges (dangerous if leaked)
SAS Tokens: Time-limited, granular permissions
Entra ID: Modern, RBAC-based (recommended for production)

Access Method Comparison:

Method Security Level Best Use Case
Access Keys Full admin Internal apps only
SAS Tokens Time-limited, granular Temporary external access
Entra ID Role-based (RBAC) Production (recommended)
✓

Step 1: View Access Keys

  1. Go to Access keys (under Security + networking)
  2. Note: two keys (key1, key2) exist for rotation without downtime
  3. Do NOT copy or share these keys (they grant full admin access)
❗

Key Security

Never commit access keys to GitHub. Always rotate keys when they're exposed.

✓

Step 2: Generate SAS Token

  1. Go to Shared Access Signature (under Security + networking)
  2. Configure these settings:
  • Allowed services: Blob
  • Allowed resource types: Container, Object
  • Allowed permissions: Read
  • Start and expiry time: Set expiry to 1 hour from now
  • Allowed protocols: HTTPS only
  1. Click Generate SAS and connection string
  2. Copy the Blob SAS URL (you'll use it in Step 3)
✓

Step 3: Test SAS Access

  1. The SAS URL you copied is formatted like:
https://<account>.blob.core.windows.net/demo-container/sample.txt?<sas-token>
  1. Paste the full Blob SAS URL into your browser
  2. You should download sample.txt without entering any credentials
  3. Note: After 1 hour, this URL will expire and access will fail
ℹ️

SAS Best Practice

SAS tokens are perfect for temporary sharing. Generate new tokens for each user/device to minimize blast radius if a token is leaked.

✓

Step 4: Test Entra ID Access

  1. Go to the storage account > Access control (IAM) > Add role assignment
  2. Select Storage Blob Data Reader role
  3. Assign to your user account
  4. Open Containers > demo-container
  5. In the top toolbar, switch Authentication method from Access key to Microsoft Entra user account
  6. Confirm you can still view sample.txt
ℹ️

Entra ID Benefits

This uses your Entra identity + RBAC instead of account keys. This is the most secure approach for production—no keys to manage or rotate.

After Part 4

  • You've reviewed access keys (and know not to share them)
  • You generated a SAS token with limited permissions and 1-hour expiry
  • You tested the SAS token to download a blob
  • You assigned an RBAC role and verified Entra ID access works

Part 5 – Network rules

⏱️ 6 minutes
⚠️

Network Rules Caution

Network rules can lock you out if misconfigured. Always allow-list your own IP before restricting access, and test thoroughly before deploying to production.

ℹ️

Network Security Layers

Even with valid access keys/SAS tokens, network rules act as a gatekeeper. They add a powerful extra layer of security above access keys.

✓

Step 1: Enable Network Rules

  1. Go to Networking (under Security + networking)
  2. Under Public network access, select Enabled from selected virtual networks and IP addresses
  3. Under Firewall, click Add your client IP address to allow-list your current IP
  4. Click Save
✓

Step 2: Test the Restriction

  1. From your current IP, confirm you can still access the storage account (you're on the allow-list)
  2. If you can, try accessing from a different network or VPN (should fail with 403 Forbidden)
  3. Observe that having the access key alone is NOT enough—network rules now take precedence
ℹ️

Production Network Security

In production, use Service Endpoints or Private Endpoints instead of IP ranges for more robust security and VNet integration.

✓

Step 3: Revert for Lab Simplicity

  1. Go back to Networking > Public network access
  2. Select Enable from all networks
  3. Click Save (to not interfere with other labs)

After Part 5

  • You enabled network rules and restricted to your IP
  • You verified access still works from your IP
  • You tested access failure from outside rules
  • You reverted to allow all networks (for lab continuation)

Part 6 – Copy data with Storage browser

⏱️ 4 minutes
ℹ️

Storage Browser vs AzCopy

The Portal's Storage browser is convenient for small files. For large data migrations and automation, use AzCopy CLI.

✓

Step 1: Copy Blob Between Containers

  1. Go to Storage browser > Blob containers > demo-container
  2. Select sample.txt > Click Copy
  3. Create a new container demo-container-2 (or navigate to an existing empty container)
  4. Click Paste—the blob is now copied
ℹ️

AzCopy Alternative

For production data migrations, the AzCopy CLI tool offers better performance and parallelization than the Portal's Storage browser.

✓

Step 2: Download and Upload

  1. Select sample.txt > Click Download (pulls to your local machine)
  2. Edit the file locally (add a comment or change the content)
  3. Go back to Storage browser > Upload
  4. Upload the modified file (overwrite the existing one)
  5. Confirm the updated file is in the container

After Part 6

  • Blob successfully copied from demo-container to demo-container-2
  • You downloaded sample.txt to your local machine
  • You modified and re-uploaded the file
  • The container reflects the updated blob content

Part 6 Validation

  • Container, queue, table, and file share all exist
  • SAS URL grants time-limited read access to blob

Cleanup (If Needed)

If you want to delete the resources created in this lab:

# Delete storage account (includes all containers, blobs, queues, tables)
Remove-AzStorageAccount -ResourceGroupName "rg-az104-lab04" -Name "stoaz104lab" -Force

# Delete resource group (removes everything)
Remove-AzResourceGroup -Name "rg-az104-lab04" -Force

Key Takeaways

  • LRS is cheap but not HA. GRS replicates to another region for disaster recovery
  • SAS tokens are time-limited and granular—perfect for sharing data without keys
  • Network rules add another layer of security on top of access keys
  • Entra ID authentication is the modern approach—don't rely on access keys in production
  • Storage browser is convenient for small files; use AzCopy for bulk data
