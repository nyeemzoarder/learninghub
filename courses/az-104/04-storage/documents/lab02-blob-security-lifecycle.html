Lab 02 – Blob Security & Lifecycle Management

Objectives

    1. Configure blob container access levels and soft delete
    2. Implement a lifecycle management policy to transition/expire blobs
    3. Enable blob versioning and point-in-time restore
    4. Configure static website hosting on a storage account

Prerequisites

    1. Storage account from Lab 01 (or create a new one)
    2. Signed in at portal.azure.com

Estimated time

35 minutes

---

Part 1 – Setup

  1. Search for Resource groups > Create. Name: rg-az104-lab05,

region: East US. Review + create > Create.

  1. Search for Storage accounts > Create. Resource group:

rg-az104-lab05, Storage account name: stoaz104lab05,

Region: East US, Performance: Standard, Redundancy: LRS,

Account kind: StorageV2. Review + create > Create.

  1. Open the storage account > Containers > + Container. Name: data.

Part 2 – Container access level

Understanding Access Levels

| Level | Listing | Reading Blobs | Best For |

|-------|---------|---------------|----------|

| Private | ❌ | ❌ | Sensitive data |

| Blob | ❌ | ✅ | Public files (share individual URLs) |

| Container | ✅ | ✅ | Public datasets (allow listing) |

⚠️

warning

Setting to "Container" exposes all blob names and allows directory listing. Only use for non-sensitive data.

  1. Containers > select data > Change access level (top toolbar).
  2. Set Public access level to Blob (anonymous read access for blobs only).

> Note: many subscriptions have Allow Blob public access disabled at

> the account level by default. If the option is greyed out, go to the

> storage account > Configuration and set Allow Blob public access

> to Enabled first.

ℹ️

tip

Share individual blob URLs instead of setting container to public. More secure and controlled.

  1. Open data > Upload > upload a small text file named public.txt

(content: "public file").

  1. Select public.txt > copy its URL from the properties pane and open

https://.blob.core.windows.net/data/public.txt in a private/

incognito browser window to confirm anonymous read access.

  1. Set the access level back to Private (no anonymous access) afterward.

Part 3 – Enable soft delete & versioning

  1. Open the storage account > Data protection (under Data management).
  2. Enable Enable soft delete for blobs — set retention to 7 days.
  3. Enable Enable versioning for blobs.
  4. Enable Enable point-in-time restore for containers (requires versioning,

change feed, and soft delete — the Portal will enable change feed

automatically if needed).

  1. Save.

Test soft delete and versioning

  1. Go to Containers > data > Upload > upload a file named test.txt

containing v1.

  1. Upload test.txt again (same name, different content v2) with

Overwrite if files already exist checked — this creates a new blob

version.

  1. Select test.txt > Delete to delete the blob.
  2. In the data container view, toggle Show deleted blobs (top toolbar) —

test.txt should appear greyed out/marked as deleted.

  1. Select the deleted test.txt > Undelete to restore it.
  2. Select test.txt > Version history (or the ... menu > Blob

versions) to confirm at least 2 versions are listed.

Part 4 – Lifecycle management policy

  1. Open the storage account > Lifecycle management (under Data

management) > + Add rule.

  1. Details tab:

- Rule name: move-to-cool-then-delete

- Rule scope: Limit blobs with filters

- Blob type: Block blobs

  1. Base blobs tab:

- Check Move to cool storage > More than 30 days after

modification.

- Check Move to archive storage > More than 90 days after

modification.

- Check Delete the blob > More than 365 days after modification.

  1. Snapshots tab: check Delete the snapshot > More than 90 days

after creation.

  1. Filter set tab: under Blob prefix, enter data/.
  2. Add to save the rule.

Exam Tips

    1. Container access levels: Private (no anonymous access), Blob (anonymous read of blobs, no listing), Container (anonymous read + listing).
    2. Lifecycle rules act on last modified date by default (or daysAfterCreation/daysAfterLastAccessTime with access tracking enabled).
    3. Point-in-time restore requires: versioning + change feed + soft delete all enabled, and only works for block blobs.
    4. Static website content is served from the special $web container; custom domains + Azure CDN/Front Door are often layered on top.
